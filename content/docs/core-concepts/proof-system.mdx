# Proof System

The proof system exists so a site action can be reviewed later without depending on memory, screenshots, or informal reconstruction.

## What gets sealed

Depending on the deployment path, the proof chain can include:

- incoming intent
- reservation or routing decision
- dispatch command
- acknowledgement or rejection
- measured outcome
- exception path

## Why the proof chain matters

It gives operators and counterparties a way to answer:

- what did the site think was true
- what action did it take
- what happened next

## Cryptographic model

The current public model centers on:

- deterministic shaping
- SHA-256 digests
- Ed25519 signatures
- explicit verification context

The important point is not novelty. The important point is independent review.
